Phase 3C — Usability Makeover (Analysis + Initial Redesign Ideas)

This page summarizes our usability analysis of Google Looker Studio’s report creation & editing flow, and proposes multiple early redesign directions. Our analysis is grounded in four individual user studies (Phase 3B) and focuses on improving learnability, discoverability, and authoring fluency for new or occasional users.

Note (billing): Our billable-hours timesheet is submitted separately (not on the public website), per the assignment instructions.

1) Usability Analysis: Problems + Why They Matter

Evidence base (from 3B)

Across four studies, the most frequent breakdowns occurred during:

  • dataset upload / connection
  • chart editing and dataset binding
  • discoverability of key editor actions
  • layout + annotation authoring interactions

1.2 What We Tackle First (Priority)

We prioritize issues that are high frequency + high impact + block progress for first-time users:

  1. P1 + P2 (Data import + binding clarity): prevents users from working with their own data correctly.
  2. P4 (Chart editing discoverability): blocks core report-building tasks.
  3. P6 (Annotation authoring): prevents users from producing a “presentable report,” even when analysis goals are clear.
  4. P3 + P5 (Discoverability + layout legibility): reduces time wasted and confusion loops.
  5. P7 + P8 (Reliability + paper-cuts): improve trust and polish.

2) Initial Redesign Ideas (Multiple Solutions per Priority Problem)

We present multiple redesign alternatives because different clients may prefer different trade-offs (simplicity vs power, novice guidance vs expert efficiency).

2.1 P1 — Data Import: Make “Upload/Import” beginner-shaped

Option A — Add a first-class “Upload file” entry point

Design concept: When users click “Add data,” show a large primary action: Upload file (CSV/Excel), alongside connectors.
Pros: Matches novice expectations; reduces hesitation.
Cons: Adds UI surface area; may feel redundant to experienced users.

Option B — Connector screen with guided “Best match” recommendations

Design concept: Keep connectors, but show “Recommended for local files” and auto-highlight CSV upload when user’s intent suggests local upload.
Pros: Preserves connector model; improves learnability with minimal new UI.
Cons: Still requires understanding connectors; may not be as obvious as Option A.

Option C — A lightweight “Import wizard” for first report

Design concept: First-time flow: “Choose data source → confirm schema → apply to report.”
Pros: Strong scaffolding; reduces wrong turns.
Cons: Adds steps; must avoid annoying repeat users.

2.2 P2 — Dataset-to-Chart Binding: Make “what’s applied” visible

Option A — Per-chart data source label (“data pill”)

Design concept: Each chart shows a small label like Data: worldcities with a quick switch dropdown.
Pros: Immediate visibility; supports recognition over recall.
Cons: Adds visual noise; must be subtle in final mode.

Option B — Post-upload banner with explicit next step

Design concept: After upload: “Dataset added. Apply to charts?” with one-click apply.
Pros: Prevents the “upload = applied” misunderstanding; minimal UI.
Cons: Needs careful design to avoid interrupting experts.

Option C — Smart apply with confirmation + undo

Design concept: If report is still on sample data, offer: “Replace sample data with your dataset (recommended)” + Undo.
Pros: Fast success path; reduces novice errors.
Cons: Risk of wrong assumptions; must keep user control.

2.3 P4 — Chart Editing Controls: Reduce ambiguity, increase discoverability

Option A — Decompose field rows into explicit actions

Design concept: Replace overloaded click zones with labeled controls:

  • Change field
  • Edit field
  • Field info
    Pros: Eliminates guessing; reduces mis-clicks.
    Cons: Slightly more UI elements; must remain compact.

Option B — Contextual menu (right-click / “...” on chart) for common actions

Design concept: Support common desktop mental model: right-click reveals change axis, sort, chart type.
Pros: Matches user expectations; reduces hunting.
Cons: Adds another interaction layer; needs discoverability too.

Option C — Expand “Chart type” by default + rename for clarity

Design concept: Always show Chart type picker prominently; avoid collapse for novices.
Pros: Quick win; improves first-time discoverability.
Cons: Takes space; may clutter for experts.

2.4 P6 — Text & Annotation Authoring: Make it feel like Slides/PowerPoint

Option A — Bigger, clearer handles + forgiving drag target

Design concept: Text boxes have strong bounding box; drag from anywhere on box border/handle area; clear edit vs move mode.
Pros: Immediate reduction in frustration; familiar metaphor.
Cons: Might conflict with Responsive constraints unless signaled.

Option B — Dedicated “Annotation mode”

Design concept: A mode toggle: “Annotate” → click to place text; move/resize is optimized; formatting stable.
Pros: Clear mental model; fewer accidental edits.
Cons: Modes can be confusing if not designed carefully.

Option C — Responsive constraints made explicit + suggested placements

Design concept: In Responsive mode, show placement zones/columns; preview snapping; explain why snapping happens.
Pros: Makes constraints legible; reduces surprise.
Cons: Adds overlays; must be unobtrusive.

2.5 P3 + P5 — Discoverability + Layout Legibility

Templates discoverability (P3)

  • Option A: Move/feature Templates on dashboard near “Create/Blank report”
  • Option B: On “Blank report” click, show “Start from template?” suggestion
  • Option C: Add template search + “recommended templates” for new users

Layout legibility (P5)

  • Option A: When truncation occurs, show inline explanation + “Fix it” buttons
  • Option B: Add “Why is this truncated?” tooltip in style panel
  • Option C: Adaptive defaults (auto-show all categories where possible)

2.6 P7 + P8 — Reliability + Paper-cuts

Reliability (P7)

  • Autosave + visible save status
  • Crash/freeze recovery: restore last state, show meaningful error feedback

Paper-cuts (P8)

  • Auto-select default title on first click
  • Prevent accidental mid-text insertion in critical fields

3) Design Rationale (Why these changes)

Our redesign directions focus on three principles:

  1. Discoverability over “hidden power”
    Many breakdowns happened because key actions were not visible where users looked first.

  2. Make system state legible
    Users cannot confidently proceed if they can’t tell what dataset is applied, or why layout constraints happen.

  3. Align with dominant mental models
    Participants brought expectations from Google Docs/Slides/PowerPoint and desktop chart editors. When Looker Studio diverges, it should either:

    • behave similarly, or
    • explain the difference at the moment it matters.

4) Pros / Cons / Trade-offs (What we’re balancing)

  • More guidance vs visual complexity
    Adding banners, labels, and hints can help novices but risks clutter for experts.
  • Automation vs user control
    Smart apply and auto-suggestions improve speed but must be undoable and transparent.
  • Consistency vs flexibility
    Making Responsive constraints clearer might reduce surprise but could restrict freedom unless the mode boundary is explicit.

5) Future Work (Next steps)

  • Validate top redesign options with 2–3 quick usability checks (can be informal).
  • Prototype the highest-impact changes first (P1/P2/P4) as low-fidelity wireframes.
  • Add “before/after” media pairs for each version we publish.
  • Use peer feedback (Phase 3D/3E) to decide which options to keep, merge, or discard.
  • If time permits, explore an “expert mode” that collapses scaffolding.
